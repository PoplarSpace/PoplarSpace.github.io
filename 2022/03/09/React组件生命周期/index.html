React组件生命周期

引出生命周期

我们仍然是先使用一个案例引出组件生命周期的概念。

需求：定义组件实现以下功能：

  • 让指定的文本做显示/隐藏的渐变动画
  • 从完全可见，到彻底消失，耗时2s
  • 点击“不活了”按钮从界面中卸载组件

组件初始建构

首先我们创建一个组件并渲染到页面上，组件包括标题和按钮，按钮绑定卸载组件事件。

那么接下来思考，我们可以通过ReactDOM.render()将组件渲染到页面上，那么该如何将这个组件从页面删除呢

首席那我们来了解一下，我们所说的将组件渲染到页面上，实际上是一个挂载的过程，那么将组件从页面上除去叫做卸载。

官方的解释是这样的：

  • 挂载：当Clock组件第一次被渲染到DOM的时候，就为其设置一个计时器。
  • 卸载：当Clock组件被删除的时候，应该清楚计时器。
标题透明度改变

那么我们来思考一下，页面标题的样式在改变，首先想到的就是组件状态。

接下来设计一个定时器。

首先根据需求：

  • 我们需要做一个周期为2s的样式改变，那么当标题透明度从1到0变化整个过程需要两秒钟，就意味着，平均2毫秒opacity就需要减0.1，因此需要循环定时器。

  • 在opacity减到0时，需要重新从一开始，再次衰减。但是由于js进制问题，有的时候不能能够准确的得到opacity=0。这就类似于0.2+0.3!=0.5一样。因为这里的判断我们写成opacity<=0。

  • 由于该状态的改变是从组件渲染到页面就开始的，因此引入一个新的APIcomponentDidMount,他和render是兄弟关系，都是react帮我们调用的API，可以直接在类里面使用的函数。它代表着“挂载完毕”时发生的事。

    在这里我们回顾一下当我们使用React.render()将组件挂载时发生的事件：

    • 1.React解析组件标签，找到了MyComponent组件。
    • 2.发现组件是使用函数定义的，随后调用该函数，将返回的虚拟DOM转为真实的DOM，随后呈现在页面

    实际上除此之外，react还会帮我们引用componentDidMount.

这里更新透明度的时候，原那本的写法是this.setState({opacity:opacity}),当key和value相同时，我们可以采用简写this.setState({opacity})

这样我们就可以基本实现了需求。

但是，值得注意的是，当我们将组件卸载时，由于定时器没有关，因此会报出错误 Can't perform a React state update on an unmounted component,他的意思就是说找不到定时器作用的组件了（那当然了，组件都卸载了嘛）。因此我们需要多添加一个关闭定时器的动作。

在这里再引入一个APIcomponentWillUnmount,他和componentDidmount也是兄弟，都是react帮我们调用的函数。它表示组件将要卸载时发生的事情。

三个API的使用时机

  • render()：组件初始化渲染，状态更新
  • componentDidMount()：组件挂载完毕
  • componentWillUnmount()：组件将要卸载

组件挂载/更新流程

  • 组件从创建到死亡会经历一些特定的阶段。
  • React组件中包含一系列钩子函数（生命周期回调函数），会在特定时刻调用。
  • 我们在定义组件时会在特定的生命周期回调函数中做特定的工作。
组件生命周期（旧）

shouldComponentUpdate返回值是true时才会调用render.如果自己不写，底层会自动调用并返回true.相当于更新的阀门。

forceUpdate():强制更新。不更改状态中的数据强制更新页面。

componentWillReceiveRrops:组件将要接收标签属性。（实际上第一次接受不会调用，意思就是“接受新的标签属性”）

A为父组件。

总结生命周期（旧）

三个阶段：

  • 初始化阶段：由ReactDOM.render()触发——初次渲染

    • constructor()
    • componentWillMount()
    • render()
    • componentDidMount()

    一般在这个钩子中做一些初始化的事，例如：开启定时器，发送网络请求，订阅消息

  • 更新阶段:由组件内部this.setState()或父组件重新render触发

    • shouldComponentUpdate()
    • componentWillUpDate()
    • render()
    • componentDidUpDate()

  • 卸载组件：由ReactDOM.unmountComponentAtNode()触发

    • componentWillUnmount()

    一般在这个钩子做一些首尾的事，例如：关闭定时器，取消订阅消息

组件生命周期（新）

新版本与旧版本的区别：

  • componentWillMount,componentWillReceiveProps,componentWillUpdate前面都需要加上UNSAFE_前缀。

    带will的钩子除了onmount都要加前缀。

    由于react自身的发展问题，这三个钩子不在经常被使用，以后可能会被废弃。

  • 添加了两个新的钩子：getDerivedStateFromProps和getSnapshotBeforeUpdate

getDerivedStateFromProps

该方法适用于一个罕见的案例，即state的值在任何情况都取决于props.

派生状态会导致代码冗余，并使代码难以维护。少用。

getSnapshotBeforeUpdate

在更新之前获取快照，将返回的结果传递给componentDidUpdate作为其第三个参数。

此法方法不常用，但是可能会出现在UI处理中，如需要以特殊方式处理的滚动位置的聊天线程等。

官网定义：getSnapshotBeforeUpdate是最近一次渲染输出之前调用。他使得组件能在发生更改之前从DOM获取一些信息（例如：滚动的位置）。

接下来写一个案例来介绍一下这个API的用法。

  • 需求：写一个带有滚轮的新闻列表，每1s会生成一条新的新闻从列表顶部往下堆；且会停留在所指定的新闻的位置，不会因为新闻数更新而被挤下去。

  • 首先因为要改变页面渲染，自然想到了状态state，我们在状态中存储一个数组保存新闻。

  • 因为每间隔1s就要更新一次state，因此我们将这个循环定时器写进componentDidMount,在初次渲染结束后开始定时器循环。

  • render()使用map()遍历数组并渲染到页面上。

  • 当然还要设置一些样式是的滚轮出现

    这样功能基本就可以实现了，最后是最重要的功能：滚轮可以停留我当前浏览的位置。

  • 这里我们就要使用到刚才学的API了：通过getSnapshotBeforeUpdate获取当前滚轮框顶部所处整个列表高度.

    scrollHeight是整个新闻列表的高度。

  • 接下来更新“显示框”顶部处于整个列表的位置。

    componentDidUpdate接受的参数分别是更新前的props 更新前的state getSnapshoyBeforeUpdate.

    这条语句的意思是“用新的新闻列表高度减去更新前的新闻列表高度，得到更新过程中新闻列表增长的高度。用更新前的显示框所处的高度再加上新增长的高度的到新的显示框所处高度。”

总结（新）

  • 初始化阶段:由ReactDOM.render()触发——初次渲染

    • constructor()
    • getDerivedStateFromProps()
    • render()
    • componentDidMount() ======>常用

  • 更新阶段：由组件内部this.setState()或父组件重新render()触发

    • getDerivedStateFromProps
    • shouldComponentUpdate()
    • render()
    • getSnapshotBeforeUpdate
    • componentDidUpdate()

  • 卸载组件：由ReactDOM.unmountComponentAtNode()触发

    • componentWillUnmount()=======>常用
